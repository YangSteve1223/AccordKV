# 审核发现汇总

## 根因定位

**PPL = 12256 的直接原因**：

| 优先级 | 问题 | Severity | 说明 |
|--------|------|----------|------|
| 1 | KV Cache 被限制在 plen=256，超出部分 padding 为零向量 | **CRITICAL** | 第 257+ 个 token 的 attention 完全失效 |
| 2 | 所有 12 样本使用完全相同的固定文本 | **NOT A BUG** | PPL 完全相同是预期行为 |
| 3 | extract_kv 捕获静态 KV，未实现动态更新 | MAJOR | 超出 256 的 token 无正确历史 |

## Critical Bug 详情

```python
# patched_attn 中的问题代码:
if ks < seq:
    pad = torch.zeros(bs, nk, seq - ks, hd, ...)  # ← 零填充导致 attention 崩溃
    cK = torch.cat([cK, pad], dim=2)
```

当 `seq=600, ks=256`：
- 344 个 token 的 key/value 被填充为零
- `softmax(Q·0^T)·0` 导致 attention 分布崩溃
- 最终 logits 完全错误 → PPL = 12256

## 修复方案

**核心修复**：在 `patched_attn` 中限制有效序列长度为 `ks`：

```python
if seq > ks:
    effective_seq = ks
    q = q[:, :effective_seq]
    pos = pos[:, :effective_seq] if pos is not None else ...
    seq = effective_seq
```

同时在 `measure_ppl_comp` 中只计算 `plen` 范围内的 PPL。

## 结论

- **不是 SVD 压缩的问题**（SVD 会导致轻微 PPL 上升，但不会到 12256）
- **不是代码逻辑错误**（除了 padding 设计问题）
- **核心问题**：KV Cache 固定长度 + 零填充的设计无法处理超过 plen 的 token
