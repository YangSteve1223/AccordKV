**审核时间**: 2026-06-27  
**审核对象**: exp_ppl_v7.py + gpu_svd_compress_v8.py  
**实验背景**: INT4 SVD KV 压缩后对下游 perplexity 的影响测量

## 执行摘要

**审核结论**: 发现 **1 个 Critical Bug** 和 **3 个 Major 设计问题**，这些是导致 PPL = 12256 的直接原因。

### 根因定位

```
compressed PPL = 12256 的直接原因:
1. [Critical] KV Cache 被限制在 plen=256，超出部分 padding 为零 → attention 完全失效
2. [Critical] 所有 12 样本使用完全相同的固定文本 → PPL 完全相同是预期行为（不是 bug）
3. [Major] extract_kv 提取的是首次 forward 的 KV，后续 tokens 未更新
```

## Bug #1: KV Cache 固定长度导致 Attention 崩溃 [CRITICAL]

### 位置
`patched_attn` 函数，padding 逻辑

### 问题描述
```python
if ks < seq:
    pad = torch.zeros(bs, nk, seq - ks, hd, dtype=cK.dtype, device=hs.device)
    cK = torch.cat([cK, pad], dim=2)
    cV = torch.cat([cV, pad], dim=2)
```

当输入序列长度 `seq > ks`（ks = 256）时：
- 超出部分的 key/value 被 **padding 为零向量**
- 这意味着第 257+ 个 token 看到的 "历史 K/V" 全是零
- Attention 计算退化：`softmax(Q·K^T)·V` 中，与零 K 相乘的注意力权重无意义

### 数学影响
假设第 300 个 token 的 query `q_300`：
- `q_300 · k_i` for i ∈ [0, 255] → 正常值
- `q_300 · 0` for i ∈ [256, 599] → **始终为零**

这导致 attention 分布严重扭曲，模型无法正确关注历史上下文。

### 影响范围
- 输入总长度 ~600 tokens
- 只有前 256 tokens 能正常工作
- 后续 ~344 tokens 的 attention 完全失效
- PPL 计算基于这些失效的 logits

### 修复方案
**方案 A（推荐）**: 限制评估范围
```python
# 只在前 plen 个 token 上评估 PPL
effective_seq = min(seq, ks)  # 256
q = q[..., :effective_seq, :]  # 只取前 256 个 query
k_exp = k_exp[..., :effective_seq, :]  # 只取前 256 个 key
v_exp = v_exp[..., :effective_seq, :]  # 只取前 256 个 value
causal = causal[..., :effective_seq, :effective_seq]
```

**方案 B**: 实现滚动 KV Cache（更复杂，需要修改 extract/compress 逻辑）

## Bug #2: 实验设计缺陷导致 PPL 完全相同 [NOT A BUG - BY DESIGN BUT MISUNDERSTOOD]

### 问题描述
用户困惑：所有 12 样本的 compressed PPL 完全相同（12256.0），这是否意味着代码有 bug？

**答案**: 不是 bug，是实验设计的**预期行为**。

### 原因分析
用户明确说明：
> "所有 12 样本的输入文本完全相同（`make_long` 生成固定文本）"

当：
- 输入文本 **完全相同**
- KV Cache 内容 **完全相同**（相同文本 → 相同 KV）
- SVD 压缩结果 **完全相同**（相同输入 → 相同压缩）
- 模型 forward 结果 **完全相同**
- PPL 计算结果 **完全相同**

**PPL 完全相同是数学上的必然，不是 bug。**

### base PPL = 1.0 vs compressed PPL = 12256 的对比
- **base PPL = 1.0**: 模型正常 forward，使用完整动态 KV → 低 PPL
- **compressed PPL = 12256**: KV 被 pad 为零 → Attention 崩溃 → 高 PPL

## Bug #3: extract_kv 捕获的是静态 KV，未考虑动态更新 [MAJOR]

### 位置
`extract_kv` 函数

### 问题描述
```python
with torch.no_grad(): model(input_ids=ids, use_cache=False)
```

1. **问题 1**: `use_cache=False` 确保模型不使用 KV Cache，但这意味着模型每次都是完整计算。这是合理的，因为我们要捕获所有 layer 的 KV。

2. **问题 2（真正的问题）**: 提取的是**首次 forward** 的 KV，但 `measure_ppl_comp` 中模型会**再次 forward**。这两次 forward 的 KV 应该相同（因为 `use_cache=False`），但如果模型内部有任何随机性或状态变化，会有差异。

### 更深层的问题：KV Cache 的语义误解

在正常推理中：
```
Token 1: 计算 K1, V1 → 保存到 Cache
Token 2: 计算 K2 = RoPE(K2_raw), 从 Cache 取 K1 → Attention(K2, [K1])
Token 3: 计算 K3, 从 Cache 取 [K1, K2] → Attention(K3, [K1, K2])
```

但在当前实现中：
```
提取 KV: [K1_raw, K2_raw, ..., K256_raw] (pre-RoPE)
使用 KV: 
  - 提取时不做 RoPE（正确，RoPE 在 attention 时应用）
  - patched_attn 中对 k_exp 应用 RoPE（正确）
```

**这实际上是正确的方式**，但问题在于：

1. 压缩后的 KV 只包含前 256 个 token
2. 模型要处理 ~600 tokens
3. 超出 256 的部分无法获得正确的历史信息

## Bug #4: RoPE 应用时机存在潜在问题 [MINOR]

### 位置
`patched_attn` 函数，RoPE 应用部分

### 问题描述
```python
# 从压缩的 KV cache 取出的 K 是 (bs, nk, ks, hd) = (1, 8, 256, 128)
# 这是 pre-RoPE 的 K
k_exp = cK.transpose(1, 2)  # (1, 8, 256, 128) → (1, 256, 8, 128)
# ... expand to num_q_heads ...
# 然后应用 RoPE
if rmode == "direct":
    qr, kr = apply_rotary_pos_emb(q.transpose(1, 2), k_exp, cos, sin, pos)
```

### 分析
1. 提取的 K 确实是 pre-RoPE（来自 k_proj 输出）
2. RoPE 在 patched_attn 中正确应用
3. **但是**：RoPE 需要知道正确的位置信息 `pos`

```python
if pos is None:
    pos = torch.arange(seq, device=hs.device).unsqueeze(0).expand(bs, -1)
cos, sin = rotary(hs, pos)
```

这里 `seq` 是当前输入的序列长度（~600），而 `k_exp` 只有 256 个位置。

### 问题
当 `seq = 600` 时，`cos, sin` 是为 600 个位置生成的。
但 `k_exp` 只有 256 个位置。

调用 `apply_rotary_pos_emb(q, k_exp, cos, sin, pos)` 时：
- `q` 有 600 个位置
- `k_exp` 只有 256 个位置
- `cos, sin` 有 600 个位置

**如果 `apply_rotary_pos_emb` 内部做了位置对齐，这可能没问题。**
**但如果没做对齐，这会导致 RoPE 应用错误。**

### 修复建议
```python
# 确保 cos/sin 与 k_exp 的 seq 维度匹配
if ks < seq:
    cos = cos[..., :ks, :]
    sin = sin[..., :ks, :]
    pos = pos[..., :ks]
```

## Bug #5: SVD 压缩对 Attention 有效性的影响 [DESIGN ISSUE]

### 问题描述
对 [T=256, D=128] 矩阵做 rank-8 SVD 压缩：
- 原始自由度: 256 × 128 = 32,768
- 压缩后自由度: 256×8 + 8 + 8×128 = 3,232（仅 9.9%）

### 理论分析
Rank-8 压缩会丢失高频成分。对于 RoPE 使用的正弦/余弦基：
- 低频成分被保留
- 高频细节被丢弃

### 对 Attention 的影响
```python
# 压缩后的 K 失去了原始 K 的高频信息
K_compressed = U @ diag(S[:8]) @ Vh[:8]
# Attention 分数: Q @ K^T
# 压缩后的 K^T 会导致 Q·K^T 的精度下降
```

### 实验观察
- 如果 PPL 略有上升（如 1.0 → 1.5），这是 SVD 压缩的正常效果
- 但 PPL 从 1.0 → 12256 远超正常范围，说明问题不在 SVD 本身

**结论**: SVD 压缩会引入精度损失，但不是 PPL = 12256 的主因。Padding 问题才是。

## Bug #6: decompress_kv 的张量形状问题 [NEEDS VERIFICATION]

### 位置
`gpu_svd_compress_v8.py` 的 `decompress_kv` 函数

### 代码
```python
def decompress_kv(comp, quantize=True, int4=True):
    K_Vt = comp["K_Vh"].float()  # [1,1,8,128] FP16→FP32
    V_Vt = comp["V_Vh"].float()
    K_U = comp["K_U"].float(); S_K = comp["K_S"].float()
    K = torch.matmul(K_U * S_K.unsqueeze(2), K_Vt)  # → [1,1,256,128]
    V = torch.matmul(V_U * S_V.unsqueeze(2), V_Vt)
    return K, V
```

### 潜在问题
1. `S_K.unsqueeze(2)` 的位置：如果 `K_U` 是 `(1, 1, 256, 8)`，`S_K` 是 `(1, 1, 8)`
2. `S_K.unsqueeze(2)` 变成 `(1, 1, 1, 8)`
3. `K_U * S_K.unsqueeze(2)` = `(1, 1, 256, 8)` × `(1, 1, 1, 8)` = broadcasting → `(1, 1, 256, 8)`
4. `matmul((1,1,256,8), (1,1,8,128))` = `(1, 1, 256, 128)` ✓

**这个逻辑看起来正确。**

### 需要验证的点
1. `compress_kv_full` 返回的字典键名是否与这里使用的一致
2. `V_U`, `S_V` 的变量名是否在代码中定义（V_U 和 S_V 是新引入的吗？）

## 完整修复方案

### 修复 1: patch_fn 中的序列长度裁剪 [CRITICAL]

```python
def patched_attn(attn_self, **kw):
    _hs = kw.get('hidden_states')
    hs  = _hs if _hs is not None else kw['hs']
    pos = kw.get('position_ids')
    # ... 原有代码 ...
    
    cK = attn_self._ck.to(hs.device)
    cV = attn_self._cv.to(hs.device)
    nk = cK.shape[1]   # num_kv_heads
    ks = cK.shape[2]   # kv sequence length
    
    q = attn_self.q_proj(hs)
    bs, seq = hs.shape[0], hs.shape[1]
    
    # === 新增: 处理 seq > ks 的情况 ===
    if seq > ks:
        # 方案 A: 只在 KV cache 范围内计算（推荐）
        effective_seq = ks
        q = q[:, :effective_seq]  # 只取前 ks 个 query
        # pos 也需要裁剪
        if pos is not None:
            pos = pos[:, :effective_seq]
        else:
            pos = torch.arange(effective_seq, device=hs.device).unsqueeze(0).expand(bs, -1)
        seq = effective_seq
    # === 修复结束 ===
    
    # ... 其余代码保持不变 ...
```

### 修复 2: measure_ppl_comp 中的 PPL 计算范围 [CRITICAL]

```python
def measure_ppl_comp(model, ids, comp_K, comp_V, plen, rope_mode):
    rotary_emb = model.model.rotary_emb
    patch_fn = make_patched_attn()
    orig = []
    
    for li, layer in enumerate(model.model.layers):
        attn = layer.self_attn
        orig.append(attn.forward)
        attn._ck = comp_K[li]; attn._cv = comp_V[li]
        attn._re = rotary_emb; attn._rm = rope_mode
        def mp(a):
            def p(**kw):
                return patch_fn(a, **kw)
            return p
        attn.forward = mp(attn)
    
    try:
        with torch.no_grad():
            out = model(input_ids=ids)
            lg = out.logits if hasattr(out, 'logits') else out[0]
            
            # === 修复: 只计算 plen 范围内的 PPL ===
            # shift_lg: 预测的是 plen+1 到 2*plen 的 token
            # shift_lb: 真实的 plen+1 到 2*plen 的 token
            effective_plen = min(plen, lg.shape[1] - 1)
            shift_lg = lg[:, :effective_plen].contiguous()
            shift_lb = ids[:, 1:effective_plen+1].contiguous()
            loss = F.cross_entropy(shift_lg.view(-1, shift_lg.size(-1)), shift_lb.view(-1), reduction='mean')
            # === 修复结束 ===
            
        ppl = torch.exp(loss).item()
    finally:
        # 恢复原始 forward
        for li, layer in enumerate(model.model.layers):
            layer.self_attn.forward = orig[li]
            for x in ['_ck', '_cv', '_re', '_rm']:
                if hasattr(layer.self_attn, x): delattr(layer.self_attn, x)
    return ppl
```
